## 💡 Executive Summary
The Recommendation Serving & Analytics Platform solves the **"latency vs. scale"** trade-off. Traditional methods of querying large datasets (Athena/SQL) often result in multi-second latencies.

This platform utilizes an **In-Memory Cache-Aside** architecture. The serving layer maintains a pre-processed mirror of the S3 data source in RAM, enabling **sub-10ms response times** while providing a rich analytical dashboard for model monitoring and governance.

## 🏁 Getting Started

### 1. Prerequisites
- Python 3.12+
- Node.js 18+
- MongoDB (Local or Atlas)
- AWS Credentials (configured via `.env` or IAM role)

### 2. Backend Setup
```bash
# Navigate to backend
cd Analytical-Platform/backend

# Create virtual environment
python -m venv venv

# Activate venv (Windows)
.\venv\Scripts\Activate.ps1

# Install dependencies
pip install -r requirements.txt

# Configure Environment
cp .env.example .env  # Update with your AWS/Mongo keys

# Seed the Database (Initial Admin setup)
python seed_db.py
```

### 3. Frontend Setup
```bash
# Navigate to frontend
cd Analytical-Platform/frontend

# Install dependencies
npm install

# Start the development server
npm run dev
```

### 4. Running the Project
- **Backend**: `uvicorn main:app --reload --workers 1`
- **Frontend**: Access via `http://localhost:5173`
- **Default Login**: `admin` / `admin123`

## 🏗 System Architecture

### High-Level Design
The system is split into three distinct tiers:
1.  **Storage Tier (AWS S3)**: The immutable source of truth. Data is stored as Parquet/CSV files generated by SageMaker.
2.  **Serving Tier (FastAPI)**: Loads the entire recommendation set into optimized memory structures.
3.  **Presentation Tier (React)**: Modular dashboard with high-frequency polling for real-time updates.

### The "Load-to-Memory" Strategy
Instead of querying a database per request, the backend performs a **Bulk Load** during startup:
- **Time Complexity**: $O(N)$ during startup load.
- **Lookup Complexity**: $O(1)$ constant time for customer retrieval via hash mapping.

## 🔍 Deep Dive Technical Details

### Data Loading Engine (`data_loader.py`)
Loaded data is stored in a `defaultdict(list)` structure, allowing instantaneous retrieval by `customer_id`.
```python
_lookup = {
    "CUST_123": [
        {"product_id": "P1", "score": 0.95, "lift": 4.2},
        {"product_id": "P2", "score": 0.88, "lift": 3.1},
    ]
}
```

### Stats Computation
To keep the dashboard lightning-fast, all aggregates (Quality Mix, Lift Distributions, Category Stats) are computed **once** during the S3 load process and served from memory.

## 🔌 API Reference

| Endpoint | Method | Purpose | Complexity |

## 📊 Model Governance

- **Quality Mix**: Tracks the balance between **Association-based** (High Signal) and **Fallback** (Generic) recommendations.
- **Model Health**: Tracks **Silhouette Scores** to monitor the clustering quality of the underlying model.
- **Segment Analysis**: Real-time filtering to see product performance across specific market segments.

## ⚙️ Operational Guide

### Environment Variables
| Variable | Description | Default |
| :--- | :--- | :--- |
| `MOCK_MODE` | Use synthetic data instead of S3 | `true` |
| `S3_BUCKET` | AWS Bucket for recommendations | `ipre-prod-poc` |
| `MONGO_URI` | Connection string for Auth DB | - |

### ⚠️ Critical Warning: Worker Count
This platform uses an **in-memory cache**. If deploying in production, you **MUST** use a single worker (`--workers 1`). Multi-worker deployments require a distributed cache like Redis to maintain consistency during `/data/refresh` calls.

## 🧪 Troubleshooting
- **White Screen**: Check if S3 data is empty; ensure optional chaining in charts.
- **Slow Startup**: Switch from `.csv` to `.parquet` in S3 for 10x faster loading.
- **401 Unauthorized**: Ensure `seed_db.py` was run and the `SECRET_KEY` matches.
